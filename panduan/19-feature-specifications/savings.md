# Savings Module — Compact
Version: 1.0

**Purpose:** Savings is Twosraku's centralized student savings management module — enabling schools to record, monitor, and manage student savings transactions transparently, securely, and auditably. It's an administrative school savings system, not a banking system.

**Philosophy:** every transaction must be traceable; balances calculated automatically; financial records never lost; transparency builds trust; savings supports financial discipline.

**Primary Objectives:** manage student savings accounts; record deposits and withdrawals; maintain transaction history; generate financial reports; provide savings statistics; integrate with Student Registry.

**Scope:** Savings Dashboard, Student Accounts, Deposits, Withdrawals, Transactions, Balance History, Reports, Statistics, Settings.

**Integrated Modules:** Student Registry, Dashboard, Statistics, Reports, Notifications, Future Parent Portal.

**Navigation:** Savings → Dashboard, Student Accounts, Deposits, Withdrawals, Transactions, Reports, Statistics, Settings.

### Dashboard
- **Purpose:** overview of student savings.
- **Widgets:** Total Balance, Today's Deposits/Withdrawals, Monthly Deposits/Withdrawals, Active Accounts, Recent Transactions, Quick Actions.
- **Quick Actions:** Deposit, Withdraw, Search Student, View Transactions, Generate Report.

### Student Accounts
- **Purpose:** manage savings accounts. Each registered student automatically has one account; accounts cannot be deleted; become inactive after graduation.
- **Properties:** Student, Student Number, Class, Major, Academic Year, Current Balance, Status, Opened Date, Closed Date, Notes.
- **Status:** Active, Inactive, Graduated, Archived.

### Deposits
- **Purpose:** record savings deposits.
- **Record:** Student, Transaction Date, Amount, Payment Method, Reference Number, Description, Recorded By, Receipt Number.
- **Business Rules:** amount must be greater than zero; balance updates automatically; transactions cannot be duplicated.

### Withdrawals
- **Purpose:** record savings withdrawals.
- **Record:** Student, Transaction Date, Amount, Reason, Approved By, Recorded By, Reference Number, Receipt Number, Notes.
- **Business Rules:** amount cannot exceed current balance; approval may be required; balance updates automatically.

### Transaction History
- **Purpose:** maintain complete financial history.
- **Types:** Deposit, Withdrawal, Adjustment, Opening Balance, Correction.
- **Properties:** Transaction Number, Student, Transaction Type, Date, Amount, Balance Before/After, Description, Recorded By, Status.
- Immutable — corrections create adjustment records instead of modifying existing transactions.

### Balance Calculation
Current Balance = Opening Balance + Deposits − Withdrawals + Adjustments. Calculated automatically.

### Search & Filters
- **Search by:** Student, Student Number, Transaction Number, Receipt Number, Reference Number.
- **Filters:** Academic Year, Semester, Class, Major, Transaction Type, Date Range, Status.

### Reports
Student Balance Report, Deposit Report, Withdrawal Report, Transaction Report, Monthly Summary, Annual Summary, Outstanding Balance, Financial Summary.

### Statistics
Total Savings, Average/Highest/Lowest Balance, Monthly Deposit/Withdrawal Trend, Student Participation, Savings Growth. Generated by the Statistics Module.

### Notifications
Deposit Recorded, Withdrawal Recorded, Low Balance, Large Withdrawal, Report Generated, Future Parent Notification.

### Settings
Configure: Minimum Deposit, Maximum Withdrawal, Approval Threshold, Receipt Number Format, Currency, Transaction Categories, Adjustment Reasons.

## Cash Closing
**Purpose:** manage daily cash reconciliation for student savings transactions — ensuring physical cash matches recorded transactions in the system. Every cashier session must be opened and closed before daily operations are completed.

**Workflow:** Open Cash Session → Process Transactions → Count Physical Cash → Compare System Balance → Resolve Differences → Close Cash Session → Generate Closing Report.

### Cash Session
**Properties:** Session Number, Cashier, Opening Balance, Opening Time, Closing Time, Closing Balance, Physical Cash, Difference, Status, Notes, Verified By.
**Status:** Open, In Progress, Pending Verification, Closed, Archived.

### Opening Cash
- **Purpose:** start a new cashier session.
- **Properties:** Opening Balance, Cashier, Date, Time, Notes.
- **Business Rules:** only one active cash session per cashier; a new session cannot begin until the previous one is closed; opening balance is mandatory.

### Closing Cash
- **Purpose:** finalize daily cashier activities.
- **Properties:** System Balance, Physical Cash Count, Difference, Closing Notes, Verified By, Closing Time.
- **Difference Calculation:** Difference = Physical Cash − System Balance.
- **Difference Status:** Balanced, Over, Short.
- **Business Rules:** closing requires physical cash verification; sessions with differences require an explanation; administrators may approve or reject the closing; closed sessions become read-only.

### Cash Reconciliation
**Purpose:** compare recorded transactions with actual cash. Displays: Opening Balance, Deposits, Withdrawals, Adjustments, Expected Cash, Physical Cash, Difference, Difference Percentage.

### Cash Closing Report
Displays: Session Number, Cashier, Opening Balance, Closing Balance, Physical Cash, Difference, Transaction Count, Deposit Total, Withdrawal Total, Adjustment Total, Verification Status. Supported formats: PDF, Excel, Print.

### Cash Closing History
Maintains historical records, immutable. Properties: Session Number, Cashier, Date, Opening Balance, Closing Balance, Difference, Verified By, Status.

### Notifications
Cash Session Opened, Cash Closing Pending, Cash Closing Completed, Difference Detected, Verification Required.

### Permissions
| Role | Access |
|---|---|
| Administrator | Full Access |
| Finance Staff | Open Session, Close Session, View Reports |
| Principal | View Reports |
| Teachers | No Access |

### Audit Log
Tracks: Cash Session Opened, Cash Session Closed, Cash Difference Recorded, Cash Closing Verified, Cash Closing Rejected. Immutable.

### Performance Requirements
Open Session < 500 ms; Close Session < 1 second; Generate Report < 2 seconds.

### Final Principle (Cash Closing)
Cash Closing ensures that every recorded financial transaction is reconciled with physical cash. Financial accountability is achieved through transparent daily reconciliation, immutable history, and verifiable closing reports.
